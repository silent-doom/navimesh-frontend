# NaviMesh — Frontend Handoff

Handoff notes for continuing implementation / integration (e.g. with Gemini Code Assist or another agent) after downloading this codebase.

---

## 1. What this is

**NaviMesh** is a dual-persona urban transit intelligence app.

- **Commuter (B2C)** — `/commuter`: user uploads a photo of a disruption + city + destination → backend returns a grounded reroute plan (markdown).
- **Planner (B2B)** — `/planner`: two tabs.
  - **Text-to-SQL**: natural-language → SQL over a transit warehouse → tabular result + executive insight.
  - **Live RAG**: city + planner query → grounded answer from live civic news + retrieved source doc + confidence score.

Frontend only. Backend is a separate FastAPI service (currently `http://127.0.0.1:8000`).

---

## 2. Tech stack (as built)

- **Framework**: TanStack Start v1 (React 19, Vite 7, SSR-capable, file-based routing under `src/routes/`)
- **Styling**: Tailwind CSS v4 via `@tailwindcss/vite`, theme tokens in `src/styles.css`
- **HTTP client**: `axios` (see `src/lib/api.ts`)
- **Icons**: `lucide-react`
- **Markdown**: `react-markdown`
- **Fonts**: Space Grotesk (display) + DM Sans (body) + JetBrains Mono (labels), all via `@fontsource-variable/*`
- **UI kit**: shadcn/ui primitives available under `src/components/ui/*` (mostly unused so far)

> Note: The original brief mentioned Next.js. We shipped **TanStack Start** instead (per the earlier decision). It is Vite-based, deploys anywhere that runs a Node/edge server, and works well on GCP Cloud Run.

## 4. Backend contract (matches `src/lib/api.ts`)

Base URL: `import.meta.env.VITE_API_BASE_URL` (default `http://127.0.0.1:8000`).

### `POST /api/v1/commuter/reroute`
- Body: `multipart/form-data` — fields `city` (string), `destination` (string), `file` (image)
- Response:
  ```ts
  { status: string; city: string; destination: string; ai_reroute_plan: string /* markdown */ }
  ```

### `POST /api/v1/planner/text-to-sql`
- Body: `application/json` — `{ user_query: string }`
- Response:
  ```ts
  {
    status: string;
    generated_sql: string;
    data: Record<string, unknown>[];
    executive_insight: string; /* markdown */
  }
  ```

### `POST /api/v1/planner/live-rag-chat`
- Body: `application/json` — `{ city: string; user_query: string }`
- Response:
  ```ts
  {
    status: string;
    city_monitored: string;
    ai_response: string; /* markdown */
    retrieved_source_document: string;
    vector_confidence_score: number;
  }
  ```

**CORS**: FastAPI must allow the frontend origin (e.g. `http://localhost:3000`, `http://localhost:5173`, and the Cloud Run URL). Add `fastapi.middleware.cors.CORSMiddleware` with `allow_origins=[...]`, `allow_methods=["*"]`, `allow_headers=["*"]`.

## 5. Running locally

```bash
bun install                     # or: npm install
cp .env.example .env            # then edit VITE_API_BASE_URL if backend is elsewhere
bun run dev                     # http://localhost:3000 (Vite dev)
# in another shell, run the FastAPI backend on 127.0.0.1:8000
```

Build:
```bash
bun run build                   # production build
bun run preview                 # preview the built app
```

## 6. Deploying to GCP

Two clean options — pick one:

### Option A — Frontend + Backend both on **Cloud Run** (recommended)

1. **Backend (FastAPI)** — containerize with a `Dockerfile` (uvicorn), `gcloud run deploy navimesh-api ...`, note the HTTPS URL.
2. **Frontend (TanStack Start)** — build produces a Nitro server. Containerize with a small Node image:
   ```dockerfile
   FROM node:20-alpine AS build
   WORKDIR /app
   COPY package.json bun.lock* package-lock.json* ./
   RUN npm ci || npm install
   COPY . .
   ENV VITE_API_BASE_URL=https://navimesh-api-XXXX.a.run.app
   RUN npm run build

   FROM node:20-alpine
   WORKDIR /app
   COPY --from=build /app/.output ./.output
   ENV PORT=8080
   EXPOSE 8080
   CMD ["node", ".output/server/index.mjs"]
   ```
   Then: `gcloud run deploy navimesh-web --source . --region <region> --allow-unauthenticated`.
3. Point the frontend at the backend by baking `VITE_API_BASE_URL` at build time (Vite inlines it) **or** by adding a runtime-configurable server endpoint (see section 8 for that refactor).
4. Add the frontend URL to the backend's CORS allowlist and redeploy the backend.

### Option B — Frontend static on **Firebase Hosting / Cloud Storage + CDN**, backend on Cloud Run

Only viable if you convert the app to a fully-static build (SPA mode). Doable but you lose SSR. Prefer Option A unless SSR is not needed.

## 7. Design system (already in place — keep consistent)

- **Palette**: "Navy Trust" — slate paper substrate, navy ink, deep municipal blue primary, emerald `signal` accent. All defined as CSS variables in `src/styles.css` and exposed as Tailwind tokens (`bg-surface`, `text-primary`, `border-signal`, etc.).
- **Type**: Space Grotesk (display), DM Sans (body), JetBrains Mono (labels/eyebrows).
- **Aesthetic**: civic infrastructure / transit wayfinding. Hairline borders (`border-border`), rectilinear geometry, monospace eyebrows in ALL CAPS with `tracking-[0.14em..0.2em]`. **No gradients, glows, glassmorphism, or purple/cyan.**
- **Do not hardcode colors** (`text-white`, `bg-[#...]`). Always use tokens.

## 8. Suggested next steps (in priority order)

1. **Env-configurable API base at runtime.** Right now `VITE_API_BASE_URL` is baked at build time. For flexible Cloud Run deployment, add a server function or a `/api/config` route that returns the backend URL from `process.env.API_BASE_URL`, and have `src/lib/api.ts` fetch it once on boot.
2. **Error UX polish.** Wire 4xx/5xx responses from the backend to `sonner` toasts (already installed) in addition to the inline error blocks.
3. **Loading states.** The Live RAG call is slow (scrape → embed → retrieve). Consider a streaming endpoint on the backend + `fetch` + `ReadableStream` on the frontend to show partial output.
4. **Auth (if needed).** No auth currently. If planners need to be gated, enable Lovable Cloud (managed Supabase) or add a Google Identity-Aware Proxy in front of Cloud Run.
5. **Analytics / observability.** Add basic pageview + API-latency logging (e.g. GA4 or a light OpenTelemetry export to Cloud Logging).
6. **Empty-state → sample data.** The landing page's "Sources · 1,204 municipal" and latency numbers in `src/routes/index.tsx` are static placeholders. Wire them to a real `/status` endpoint if the backend exposes one.
7. **Mobile.** Layout is responsive but the commuter camera flow deserves QA on iOS Safari (the `capture="environment"` attribute is already set on the file input).
8. **SEO.** Per-route `head()` metadata is set. Add a real `og:image` (hosted at an absolute HTTPS URL) on `__root.tsx` — currently omitted.

## 9. Known gotchas

- **TanStack Router**: never edit `src/routeTree.gen.ts`. Add a file under `src/routes/`, and the plugin regenerates it on next build/dev.
- **Server-only env**: `process.env.*` is only defined on the server (server functions, SSR). Anything the browser reads must be `import.meta.env.VITE_*`.
- **Tailwind v4**: theme tokens live in `src/styles.css` under `@theme inline`, not in a `tailwind.config.js`. Do not add a v3-style config file.
- **Nominatim reverse geocoding** is used in `commuter.tsx` for the "Detect" button. It is a free public service with strict rate limits — replace with Google Maps Geocoding API before production traffic.
- **File uploads**: axios sets `Content-Type: multipart/form-data` explicitly; do not override with a JSON header on that route.
